## Why These Two Disciplines Belong Together

Most conversations treat AI development and DevOps as separate tracks. Train your model over here. Set up your pipelines over there. But the moment you start deploying AI-powered automation at any meaningful scale, the separation breaks down.

An n8n workflow that calls an LLM is a service. It needs:

- A runtime environment (container)
- Environment variable management (secrets)
- Uptime monitoring (health checks)
- Version control for prompts (yes, prompts are code)
- A rollback path when the model starts hallucinating

That's a DevOps problem.

## Prompts as Code

This is the insight I keep coming back to: **prompts are configuration files.** They should be versioned, reviewed, and deployed — not hardcoded in a database field or pasted into a UI.

My current approach: prompts live in `.txt` files in a `/prompts` directory, committed to the repo, injected into containers as environment variables or mounted volumes. That way, a prompt change is a PR, not a manual database update.
